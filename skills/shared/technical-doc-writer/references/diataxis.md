# Diataxis 分類指引

使用這份參考文件來判斷技術文件的主要類型，避免一份文件同時扮演多個角色。

## 先問的問題

先判斷讀者此刻最想完成的是哪一件事：

- 想學會某個系統或功能的基本使用方式
- 想立刻完成某個具體任務
- 想理解背後概念、架構與設計取捨
- 想查詢精確規格、指令、參數或欄位

答案會直接對應到文件類型。

## 1. Tutorial

目標：帶新手一步一步做出成果。

適合：

- 第一次接觸某技術或系統
- onboarding
- Hello World
- 第一個成功案例

應該包含：

- 明確的學習成果
- 逐步操作
- 預期結果
- 能快速獲得小成功的路徑

不應該包含：

- 大量架構推導
- 過度完整的參數百科
- 太多分支情境

判斷提示：

- 如果讀者需要有人牽著走，這通常是 Tutorial

## 2. How-to

目標：讓讀者完成單一任務。

適合：

- 部署服務
- 新增設定
- 修復常見問題
- 執行遷移
- 串接某個外部系統

應該包含：

- 任務目標
- 前置條件
- 具體步驟
- 驗證方式
- 常見錯誤與排除方式

不應該包含：

- 大量理論背景
- 與任務無關的歷史脈絡
- 廣泛的全面性介紹

判斷提示：

- 如果讀者只是想把事情做完，這通常是 How-to

## 3. Explanation

目標：說明背後的為什麼。

適合：

- 架構設計
- 技術選型
- 系統限制
- trade-offs
- 領域概念

應該包含：

- 問題背景
- 設計思路
- 架構圖或流程圖
- 取捨理由
- 替代方案與限制

不應該包含：

- 假裝是操作手冊的步驟列表
- 詳細到百科全書等級的規格清單

判斷提示：

- 如果核心問題是「為什麼這樣設計」，這通常是 Explanation

## 4. Reference

目標：讓讀者快速查到精確資訊。

適合：

- API 文件
- CLI 指令表
- 設定檔欄位說明
- schema 與參數清單

應該包含：

- 穩定且一致的欄位結構
- 表格
- 預設值與限制
- 範例輸入輸出

不應該包含：

- 冗長故事
- 大段背景說明
- 教學式引導

判斷提示：

- 如果讀者是來查資料，不是來上課，這通常是 Reference

## 常見混淆

### Tutorial vs How-to

- `Tutorial` 是教新手學會
- `How-to` 是幫使用者完成任務

若重點是學習路徑，用 Tutorial。
若重點是任務完成，用 How-to。

### How-to vs Explanation

- `How-to` 重點是做法
- `Explanation` 重點是原因

如果背景知識會妨礙任務流暢度，就把它拆出去，另外寫成 Explanation。

### Explanation vs Reference

- `Explanation` 重點是理解
- `Reference` 重點是查詢

如果內容是參數、欄位、限制、回傳格式，優先用 Reference。

## 實務分類原則

- 一份文件只選一個主類型
- 可以連到其他類型文件，但不要在同一份文件裡混寫
- 若文件有兩個以上主要目的，通常表示應拆成多份文件
- 任務型文件可以連到概念型文件，但不要要求讀者先讀完整理論才能完成任務
