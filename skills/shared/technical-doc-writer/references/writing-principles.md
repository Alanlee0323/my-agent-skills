# 技術文件寫作原則

這份文件整理技術文件的核心寫法，用來指導內容取捨與語氣控制。

## 1. 先知道讀者是誰

寫之前先判斷讀者：

- 同組工程師
- 跨組工程師
- PM
- 營運或支援角色
- 高階主管

不同讀者需要的內容不同：

- 工程師需要實作細節、指令、程式碼、架構脈絡
- PM 需要流程、限制、依賴關係、風險與影響
- 管理層需要摘要、圖表、決策影響與取捨

不要用同一份文件同時滿足所有人。若受眾差異太大，拆文件通常比加章節更有效。

## 2. 一份文件只做一件事

完成草稿後，檢查每一段是否支援同一個目標。

如果某段內容不支援主目標，就：

- 刪掉
- 拆到另一份文件
- 用連結導向相關文件

技術文件最常見的問題不是寫太少，而是混入太多不該在這份文件裡的內容。

## 3. 為掃讀而寫

讀者通常不會逐字閱讀。請假設他們會：

- 先看標題
- 看段落第一句
- 找 code block
- 找表格
- 找關鍵字

因此：

- 最重要的資訊放在前面
- 標題要能自我解釋
- 段落開頭先講結論
- 長段落切短
- 優先用列表、表格、步驟與區塊

## 4. 不預設立場

不要假設讀者已經：

- 裝好環境
- 具有權限
- 知道服務名稱與背景
- 理解你團隊的內部術語

凡是讀者需要事先知道的事情，都應該明確寫出來，或提供連結。

技術文件不必過度追求 DRY。對不同文件重複關鍵上下文，通常比逼讀者跳來跳去更好。

## 5. 展示，不要只敘述

能展示就不要只講抽象文字。

優先使用：

- 範例指令
- 範例輸入輸出
- 範例程式碼
- Mermaid 或 PlantUML 圖
- 流程圖
- 表格

對新手尤其重要的是「最快獲得第一個成功結果」的路徑。

## 6. 使用簡單直白的中文

即使讀者是技術人員，也不代表文件需要寫得艱澀。

建議：

- 句子短
- 主詞清楚
- 避免空泛形容詞
- 能直接下結論就不要繞
- 中英文混用時，以中文敘述為主，保留必要專有名詞英文

不建議：

- 過度翻譯已廣泛使用的技術名詞
- 把一句話拆成太多抽象修飾
- 用像簡報官話的句型包裝簡單意思

## 7. 內容順序要符合使用頻率

最常被查的資訊應該放前面。

例如：

- 常用指令放前面
- 常見情境放前面
- 常見錯誤放前面
- 冷門 edge case 放後面或附錄

若讀者三次點擊內仍找不到內容，通常代表資訊架構需要重整。

## 8. Docs as Code

技術文件應優先以 Markdown 維護，並與程式碼一起進版控。

實務原則：

- 用 PR review 文件
- 讓文件更新成為 Definition of Done 的一部分
- 用 CI 檢查失效連結、格式與必要欄位
- 在文件中標註版本、日期、適用範圍或假設條件

過期文件比沒有文件更糟。

## 9. AI 的正確用法

把 AI 當成整理與轉換工具，不要盲目要求它憑空產出一篇長文。

適合交給 AI 的事情：

- 把雜亂筆記整理成 Markdown
- 把 CLI 輸出轉成表格
- 依固定模板補齊骨架
- 協助分類 Diataxis 類型
- 檢查文件是否混入非目標內容

不適合直接接受而不檢查的事情：

- 憑空捏造架構細節
- 產出沒有上下文驗證的操作步驟
- 冗長但資訊密度很低的段落
